**Primary Issue**: `FirstVisitSplash` component blocks ALL web UI tests by overlaying viewport (`z-index: 60`, covers entire screen after 800ms delay).

**Secondary Issue**: `normalizeHttpUrl()` auto-prepends `https://` which resolves path traversal and allows non-http schemes to slip through normalization.

## Solution Plan

### Phase 1: Fix API URL Validation (packages/shared)

**File**: `packages/shared/src/lighthouse/lib/validation.ts`

Enhance `normalizeHttpUrl()` function:

```typescript
export function normalizeHttpUrl(input: unknown): string | null {
  if (typeof input !== "string") {
    return null;
  }

  const raw = input.trim();
  if (!raw || raw.length > 2048 || /[\r\n]/.test(raw)) {
    return null;
  }

  // Reject path traversal attempts before normalization.
  if (raw.includes("../") || raw.includes("..\\")) {
    return null;
  }

  // If the input explicitly specifies a protocol, it must be http/https.
  // This prevents ftp://, file://, data:, javascript:, etc.
  const hasProtocol = /^[a-zA-Z][a-zA-Z0-9+.-]*:/.test(raw);
  if (hasProtocol && !/^https?:\/\//i.test(raw)) {
    return null;
  }

  const candidate = /^https?:\/\//i.test(raw) ? raw : `https://${raw}`;

  try {
    const url = new URL(candidate);
    if (!["http:", "https:"].includes(url.protocol) || !url.hostname) {
      return null;
    }

    url.hash = "";
    return url.toString();
  } catch {
    return null;
  }
}
```

**Changes**:
1. Added path traversal check (`../` and `..\\`)
2. Added protocol validation: reject non-http(s) explicit protocols

Note: The ftp test currently returns 200 (unexpected). The enhanced validation should explicitly reject `ftp://` via the new protocol check. If the test still fails, investigate further - possibly the request body format or a different code path.
